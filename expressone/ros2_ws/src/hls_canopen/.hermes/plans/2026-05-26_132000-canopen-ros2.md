# canopen_ros2 — HLS CANopen WebSocket → ROS2 桥接

## 目标

新建 **独立 Rust 包** `canopen_ros2`，通过 WebSocket 连接已有 `hls_canopen` 服务器（ws://host:9090），桥接到 ROS2 话题。

**不修改 hls_canopen 一行代码**。hls_canopen 继续做纯 CAN 控制 + WebSocket 接口，`canopen_ros2` 只管 ROS2 桥接。

---

## 架构

```
                    ┌─ our new package ─────────────────────┐
                    │                                        │
                    │  canopen_ros2 (Rust + r2r)             │
Nav2 ──sub──← /cmd_vel     │  ──→ WS "linear/angular"       │
     ──pub──→ /odom        │  ←── WS "pose/status"          │
     ──pub──→ /tf          │  ──→ WS "ping" keepalive       │
                    │                                        │
                    └──────────┬──── WebSocket ──────────────┘
                               │ ws://192.168.8.30:9090
                               ▼
                    hls_canopen (纯 CAN 控制)
                    CANopen → DS20270DA 伺服
                    can0, 1 Mbps, node 0x01/0x02
```

## WebSocket 协议

**发送**（字符串 JSON，20 Hz + 有指令时立即发送）：
```json
{"linear": 0.5, "angular": 0.1}
```

**接收**（20 Hz，与 hls_canopen 控制循环同步）：
```json
{
  "linear": 0.390,
  "angular": 0.090,
  "left_rpm": 450,
  "right_rpm": -442,
  "pose": {
    "x": 0.1200,
    "y": 0.0150,
    "theta": 0.0500
  },
  "status": "enabled"
}
```

**故障时**：
```json
{
  "error": "CAN down",
  "status": "disabled",
  "pose": {
    "x": 0.1200,
    "y": 0.0150,
    "theta": 0.0500
  }
}
```

## 话题设计

### 订阅

| 话题 | 类型 | 说明 |
|------|------|------|
| `/cmd_vel` | `geometry_msgs/Twist` | `.linear.x` (m/s), `.angular.z` (rad/s) → 发送到 WebSocket |

### 发布

| 话题 | 类型 | 频率 | 说明 |
|------|------|------|------|
| `/odom` | `nav_msgs/Odometry` | 20 Hz | 从 WS pose 构建 |
| `/tf` | `tf2_msgs/TFMessage` | 20 Hz | `odom` → `base_link` 变换 |

### 帧 ID

| 字段 | 值 |
|------|-----|
| `/odom.header.frame_id` | `"odom"` |
| `/odom.child_frame_id` | `"base_link"` |
| `/tf.transform.header.frame_id` | `"odom"` |
| `/tf.transform.child_frame_id` | `"base_link"` |

## 项目结构

```
~/ros2_ws/
└── src/
    └── canopen_ros2/             ← 独立 Rust 包
        ├── Cargo.toml
        ├── src/
        │   └── main.rs           ← r2r 节点 + WebSocket 客户端
        └── launch/
            └── canopen_ros2.launch.py
```

### main.rs 核心逻辑

```
init:
  1. clap 解析参数 (--ws-url, --frame-id, --pose-covariance, --twist-covariance)
  2. 创建 r2r Context + Node ("canopen_ros2")
  3. 创建 /cmd_vel subscriber
  4. 创建 /odom publisher
  5. 创建 /tf publisher
  6. WebSocket 连接到 hls_canopen (ws://192.168.8.30:9090)

loop (20 Hz):
  1. spin_once(Duration::from_millis(1))
  2. 如果有新的 cmd_vel：{"linear": x, "angular": z} 发送到 WS
  3. WS 接收：解析 JSON → pose + status
  4. 如果有 error（"CAN down"）→ log 警告，pose 冻结
  5. pose (x,y,theta) → Odometry 消息
     - pose.position = {x, y, 0}
     - pose.orientation = Quaternion from yaw
     - twist.linear.x = linear (来自 JSON)
     - twist.angular.z = angular (来自 JSON)
  6. pose → TFMessage (odom→base_link)
  7. 发布 /odom 和 /tf
  8. 断开后：等待 1s 自动重连，不退出
```

### `/cmd_vel` 回调逻辑

```rust
// 收到 Twist 消息 → 更新 shared state
// 控制循环在每个周期检查 shared state
// 如果有新指令→发送到 WS
// 如果 1s 没收到指令→发送 {"linear":0,"angular":0}
```

### 连接管理

- 启动时连接，断线时自动重连（指数退避 1s→2s→4s→max 30s）
- 重建连接后，重新发送当前速度值
- 断线期间 `/odom` 继续以最后已知 pose 发布（冻结），TF 不变
- 日志记录连接状态变化

## 分步计划

### Step 1: 创建项目

```bash
mkdir -p ~/ros2_ws/src/canopen_ros2/src
cd ~/ros2_ws/src/canopen_ros2
cargo init --name canopen_ros2
```

### Step 2: 编写 Cargo.toml

依赖：
- `r2r = "0.9.5"` — ROS2 桥接
- `tungstenite = "0.26"` — WebSocket 客户端
- `serde = { version = "1", features = ["derive"] }` — JSON 解析
- `serde_json = "1"` — JSON 序列化
- `url = "2"` — URL 解析
- `clap = { version = "4", features = ["derive"] }` — CLI 参数
- `log = "0.4"` — 日志

### Step 3: 编写 main.rs

~250 行，核心逻辑如上。

### Step 4: 编译

```bash
zsh -c 'source ~/.zshrc; cd ~/ros2_ws/src/canopen_ros2 && cargo build'
```

### Step 5: 创建启动脚本

```bash
~/canopen_ros2.sh
```

类似 `~/keli_lidar.sh` 风格，带参数传递和 ROS2 环境检查。

### Step 6: 验证

```bash
# 终端 1: hls_canopen (CAN 控制)
# 终端 2: canopen_ros2
zsh -c 'source ~/.zshrc; ~/ros2_ws/src/canopen_ros2/target/debug/canopen_ros2 --ws-url ws://127.0.0.1:9090'

zsh -c 'source ~/.zshrc; ros2 topic pub /cmd_vel geometry_msgs/Twist "{linear: {x: 0.5}, angular: {z: 0.0}}" --once'
```

### Step 7: 推送到远程仓库

## 关键差异 vs 直接集成（之前方案）

| 维度 | 之前方案（hls_canopen 内加 binary） | 新方案（独立包 canopen_ros2） |
|------|------|------|
| 包依赖 | 依赖 hls_canopen 全部 CAN 代码 | 零 CAN 依赖，仅 WebSocket |
| 耦合度 | 高（修改 Cargo.toml + src/bin/） | 低（完全独立） |
| 部署 | 和 CAN 驱动在同一进程 | 可独立启停 |
| 故障隔离 | hls_canopen 崩溃 → ROS2 也跟着崩 | 各崩各的 |
| 调试 | 同时看 ROS2 和 CAN 日志不方便 | 分进程，日志分离 |
| 远程部署 | 只能用同一个机器 | canopen_ros2 可远程运行（跨机器） |

## 风险和待确认

1. **r2r 的 tf2_msgs/TFMessage 支持** — 需要确认 r2r 是否内置这个 msg 类型。如果不支持，降级为手动发布 `/tf` 话题（直接用 raw 字符串 publish），Nav2 只认标准 TFMessage 格式
2. **WebSocket 消息频率同步** — hls_canopen 以 20 Hz 推送 odom，canopen_ros2 也是 20 Hz 循环，两者天然同步
3. **CmdVel → WebSocket 的时机** — 两个策略选一个：
   - (a) 收到 cmd_vel 回调**立即**发 WS（延迟最小）
   - (b) 控制循环里**周期性地**发最新值（简单，但可能多一条延迟）
   - 选 **(a)**，因为这是桥接层的核心功能——实时性
4. **断线期间的 cmd_vel 积压** — 连不上时不积压 cmd_vel，断线时丢弃所有指令，重连后发送最后已知值
