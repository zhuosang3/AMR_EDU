# Nav2 集成：需要输出的两个话题

> 目标：在现有 cmd_vel 输入基础上，确定让 Nav2 正常导航还需输出的两个话题
> 日期：2026-05-23

---

## 背景

HLS_canopen 已经有 WebSocket 接收 `cmd_vel`（v/ω → RPM → CANopen SDO → 驱动器）。
要接入 ROS2 Nav2，除了能执行速度指令外，Nav2 还需要知道机器人"认为自己在哪里"。

## Nav2 必需的两个输出话题

| # | 话题 | 消息类型 | 作用 |
|---|------|---------|------|
| 1 | **`/odom`** | `nav_msgs/Odometry` | 里程计：累计位姿 (pose) 和瞬时速度 (twist) |
| 2 | **`/tf`** | `tf2_msgs/TFMessage` | 坐标变换：持续发布 `odom` → `base_link` 的变换 |

### `/odom` 内容

```
Header header
  builtin_interfaces/Time stamp       ← 数据时间戳
  string frame_id = "odom"
string child_frame_id = "base_link"
PoseWithCovariance pose               ← 累计位置 (x, y, theta)
  Pose pose
    Point position (x, y, z)
    Quaternion orientation
  float64[36] covariance
TwistWithCovariance twist             ← 瞬时速度 (vx, vy, vtheta)
  Twist twist
    Vector3 linear  (vx, vy, vz)
    Vector3 angular (vx, vy, vz)
  float64[36] covariance
```

关键字段：`pose.pose.position`（x, y）、`pose.pose.orientation`（yaw）、`twist.twist.linear.x`、`twist.twist.angular.z`。

### `/tf` 内容

本质上是连续发布 `odom` → `base_link` 的 transform，与 `/odom` 中的 pose 一致。

大多数实现不直接发 `/tf`，而是：
- 发 `/odom`（`nav_msgs/Odometry`）
- 发 `/joint_states`（`sensor_msgs/JointState`，左轮/右轮位置速度）
- 让 `robot_state_publisher` 根据 URDF 发布静态 tf（base_link→wheel等）
- 用一个 **odometry publisher 节点** 同时读 encoder 数据 → 发布 `/odom` **和** `odom→base_link` 的 tf

## 现有数据能否满足？

HLS_canopen 已有的数据源：

| 数据 | 来源 | 状态 |
|------|------|------|
| 左右轮 RPM | 0x606C SDO 读取 | ✅ 已有（read_velocity） |
| 轮径 169mm，轮距 450mm | 常量 | ✅ 已有 |
| 右轮反转标志 | 常量 | ✅ 已有 |
| 时间戳 | std::time::Instant | ⚠️ 需要改为连续单调时钟 |

**从 RPM → odom 的计算链路已经完整：**

```
左轮 RPM, 右轮 RPM
  → 左轮线速度 v_L = RPM_L × π × D / 60
  → 右轮线速度 v_R = RPM_R × π × D / 60
  → 线速度 v = (v_L + v_R) / 2
  → 角速度 ω = (v_R - v_L) / W   (W = 轮距)
  → Δx = v × cos(θ) × dt
  → Δy = v × sin(θ) × dt
  → Δθ = ω × dt
```

## 架构选项

### 选项 A：HLS_canopen 直接发 ROS2 话题（不推荐）

在 Rust 二进制里引入 ros2-client 或自己组 UDP 包发 ROS2。但：
- Rust ROS2 生态不成熟（ros2-rust 需 colcon + ament）
- musl 静态编译困难
- 引入了新的依赖栈

### 选项 B：桥接进程（推荐）

```
┌──────────────┐   WebSocket JSON    ┌──────────────────┐   ROS2 topics   ┌───────┐
│ HLS_canopen  │ ── ws://9090 ────→  │ odom_bridge.py   │ ── /odom,/tf ─→ │ Nav2  │
│ (CANopen SDO)│ ←── cmd_vel ◄────── │ ros2 node        │                │       │
└──────────────┘                     └──────────────────┘                └───────┘
```

一个轻量 Python 进程：
1. 连 WebSocket → 收 odom JSON（已有：`{"linear":...,"left_rpm":...,"right_rpm":...,"status":...}`）
2. 发 cmd_vel JSON → HLS_canopen
3. 订阅 ROS2 `/cmd_vel` → 转 JSON 发给 HLS_canopen
4. 从 odom JSON → 发布 `/odom` (nav_msgs/Odometry) + `/tf`

优点：
- HLS_canopen 保持干净（纯 CANopen，不引入 ROS2 依赖）
- Python 中 ROS2 生态完善（rclpy）
- 独立部署：ROS2 坏了不影响底层电机控制

## 实施计划

### Phase 1：扩展 odom JSON（HLS_canopen 端）

当前 odom 输出：`{"linear":...,"angular":0,"left_rpm":...,"right_rpm":...,"target_v":...,"status":"..."}`

需要新增累积位姿字段：
```json
{
  "linear": 0.39,
  "angular": 0.05,
  "left_rpm": 450,
  "right_rpm": -442,
  "target_v": 0.4,
  "status": "enabled",
  "pose": {"x": 1.23, "y": 0.05, "theta": 0.03},
  "cov": [0.001,0,0,0,0,0, 0,0.001,0,0,0,0, ...]
}
```

修改文件：`src/main.rs`
- 在控制循环中累计 `pose_x`, `pose_y`, `pose_theta`
- 每次 odom 输出时附加 pose

### Phase 2：odom_bridge Python 节点

新建文件：`/root/myfile/EXP_EDU/odom_bridge/odom_bridge.py`

功能：
1. `rclpy` 节点 `odom_bridge`
2. 订阅 `/cmd_vel` (geometry_msgs/Twist) → JSON → WebSocket 发给 HLS_canopen
3. 从 WebSocket 收 odom JSON → 发布 `/odom` (nav_msgs/Odometry)
4. 同时发布 `odom` → `base_link` 的 tf

### Phase 3：启动脚本

```bash
# 1. 启动 CANopen 控制进程
./hls_canopen &

# 2. 启动桥接
ros2 run odom_bridge odom_bridge.py &

# 3. 启动 Nav2
ros2 launch nav2_bringup navigation_launch.py
```

## 注意事项

- **odom 不是定位**：odom 只是轮式里程计，会随时间漂移。Nav2 需要 AMCL（激光+地图）做全局定位，odom 只用于短期推算
- **协方差**：`/odom` 要求填 36 个 float64 协方差矩阵。初期可以填经验值（如 vx=0.001），后续按 encoder 精度调
- **时间戳单调性**：odom 需要稳定的单调时间源。HLS_canopen 的 odom 1Hz 频率偏低，Nav2 建议至少 10-20Hz。需要提频
