# 全局路径频繁重规划修复 (2026-06-03)

## 现象
发送 goal 后不一会儿，RViz 中出现一堆全局路线叠加。

## 根因
`global_costmap` 的 `obstacle_layer` 将 `obstacle_max_range` 设为 10m，远端雷达噪点被标记为障碍物，导致 global_costmap 出现闪烁障碍物 → planner 反复重规划 → /plan 话题发布多条路径叠加。

对比 `local_costmap` 的 obstacle_max_range 只有 2.5m，全局规划本就不该操心 10m 外的噪点。

## nav2_params.yaml 改动

### global_costmap — obstacle_layer 范围收窄
| 参数 | 旧值 | 新值 | 说明 |
|------|------|------|------|
| `obstacle_layer.scan.obstacle_max_range` | `10.0` | `2.5` | 只标记 2.5m 内障碍物，与 local 一致 |
| `obstacle_layer.scan.raytrace_max_range` | `10.0` | `3.0` | 只清除 3m 内的障碍物 |
| `obstacle_layer.combination_method` | (无) | `1` | 多传感器数据取最大值（Overwrite），避免叠加噪声 |

### planner_server — 显式指定 costmap
| 参数 | 旧值 | 新值 | 说明 |
|------|------|------|------|
| `costmap_topic` | (无，默认) | `/global_costmap/costmap_raw` | 显式订阅，避免默认 topic 不匹配 |

### behavior_server — global_frame 修正
| 参数 | 旧值 | 新值 | 说明 |
|------|------|------|------|
| `global_frame` | `odom` | `map` | 恢复行为与全局路径坐标系一致，避免 backup/spin 目标漂移 |

## RViz 端配合
Path 显示 → Buffer Length 建议设为 1，只保留最新一条全局路径。

## 后续恢复
无需恢复。obstacle_max_range=2.5m 对全局规划足够，远端障碍物由 local_costmap 在接近时处理。

# planner 禁止走未知区域 (2026-06-03)

## 现象
SmacPlanner2D 规划的全局路径绕大圈走未知区域，明明直行在已知地图内就能到。

## 根因
`allow_unknown: true` 允许路径穿过未知区域，且未知区域代价中性，有时比已知区域（受 inflation_layer 影响）更"便宜"，planner 优先绕行未知区域。

## nav2_params.yaml 改动
| 参数 | 旧值 | 新值 | 说明 |
|------|------|------|------|
| `GridBased.allow_unknown` | `true` | `false` | 未知区域视为不可通过，路径只走已知地图 |

## 后续恢复
切回 `allow_unknown: true` 即可。如果建图质量好、边界清晰，true 可以接受。当前地图有大量未探索区域，false 更安全。

# 旋转漂移修复・第一批 (2026-06-11)

## 现象
导航中只要稍有旋转, AGV 定位就漂移; 直线行驶基本正常。

## 根因分析 (按嫌疑排序)
1. **RPM 整数量化**: odom 来源是 SDO 0x606C 整数 RPM。角速度限制 0.15 rad/s
   对应每轮差速仅 ±3.8 RPM, 整数读数 ±0.5 RPM 误差 → 角速度误差 ~13%;
   直线 34 RPM 时误差仅 1.5%。完美解释"直线还好, 旋转就漂"。(本批未根治,
   待第二批改用 0x6064 位置计数差分)
2. **后万向轮转向阻力**: 原地旋转时万向轮被掰转 180°, 驱动轮打滑 → theta 虚高。
3. **固定 DT=0.05 积分 bug**: 实际周期 ≈ 55~65ms (ws.read 超时 + 3 次阻塞 SDO 读),
   固定 0.05 系统性少积分 10~30%。→ 本批已修。
4. **轮距未标定**: 有效轮距受轮胎接触面/负载影响, 误差只对旋转生效。→ 本批给出标定脚本。
5. **IMU 闲置**: `amcl.imu_topic` 是无效参数 (nav2_amcl 无 IMU 接口, 被静默忽略),
   /imu 话题零消费者。→ 本批删掉无效参数, EKF 融合留待第二批。

## hls_canopen/src/main.rs 改动 (改完已 cargo build/test 通过)
| 项 | 旧 | 新 | 说明 |
|------|------|------|------|
| 积分时间步 | `const DT: f64 = 0.05` 固定 | `last_odom.elapsed()` 实测 | 消除 10~30% 系统性少积分 |
| dt 上限 | 无 | `dt.min(0.2)` | 重连/卡顿后不把大间隔当一步积分 |

## nav2_params.yaml 改动
| 参数 | 旧值 | 新值 | 说明 |
|------|------|------|------|
| `amcl.alpha1` | `0.2` | `0.4` | 旋转→旋转噪声调大, 旋转时少信里程计多信激光 |
| `amcl.update_min_a` | `0.2` | `0.1` | 0.2 rad≈11.5° 才更新粒子, 旋转中外推太久; 降到 ~5.7° |
| `amcl.transform_tolerance` | `1.0` | `0.5` | map→odom 修正滞后 1s 太久, 旋转时表现为漂移 |
| `amcl.imu_topic` | `/imu` | (删除) | nav2_amcl 没有 IMU 接口, 此参数无效, 删除以免误导 |

## 轮距标定 (待人工执行)

# 修正: RAD_PER_COUNT 锁定 5600 counts/rev (2026-06-11 晚)

## 问题
修正 0x6063 后，代码用 TAU/10000（基于 EDS 默认 10000 counts/rev），
但用户手动推轮一圈测得 delta=5608 counts，与 10000 不符。

## 根因
电机型号 **10530NA1550F20**，按 `电机手册.pdf` 命名规则解码：

| 段位 | 值 | 含义 |
|------|-----|------|
| 105 | 105 | 定子外径 105mm |
| 30 | 30 | 定子厚度 30mm |
| N | N | 24V |
| **A** | **A** | **编码器类型: 1400 线光编** |
| 15 | 15 | 最大转矩 15N·m |
| 50 | 50 | 最高转速 500rpm |
| F | F | 极对数 15 |
| 2 | 2 | TPE 轮胎 |
| 2 | 2 | 设计代码 |

之前误以为是 "D" 型编码器（2500 线 → 10000 counts/rev），
实际为 "A" 型：**1400 线 × 4 倍频 = 5600 counts/rev**。
实测 5608 与理论 5600 仅差 0.14%（手动推轮测量误差）。

**SDO 全部不暴露**: 0x608F.1(encoder resolution)、0x2006(Fn0.006)、
0x2007(Fn0.007)、0x6091(gear ratio) 均返回 abort 或超时。
DS20270DA 不通过 CANopen 暴露这些参数，只能靠电机型号反查。

## 改动
| 文件 | 改动 |
|------|------|
| src/kinematics.rs | RAD_PER_COUNT: TAU/10000.0 → **TAU/5600.0** (含注释) |
| src/kinematics.rs | TEST_RAD_PER_COUNT: TAU/10000.0 → TAU/5600.0 |

## 影响
**里程计数据系统性差 1.78×（之前所有数据偏低 44%）**，波及：
- 线速度/角速度 → 全部偏低
- EKF 线速度无第二传感器纠正 → odometry/filtered 同样偏低
- AMCL 运动模型预测 → 粒子落后真实位置，激光强行拽回 → 定位跳变
- 导航控制 → 实际超速 1.78×，接近目标时过冲震荡

此次修正后以上全部解决。

## 验证
- cargo build + 35 测试全过
- 动车实测仍需按"待实测"清单执行（线速度和定位应该大幅改善）

# 修正: 编码器计数方向 (2026-06-12)

## 现象
导航模式下往前推车，地图上小车向后退。

## 根因
DS20270DA 的 0x6063（位置反馈）在电机**电气正转**时是 **DECREASE**（减计数），
而非之前假设的 INCREASE（增计数）。

`position_delta_to_displacement` 里：
```
arc_l = d_left_counts * RAD_PER_COUNT * r   // +counts → 正值
```
但电气正转时 d_left_counts 实际为负 → arc_l 为负 → d_linear 为负 → 后退！

数学推导 (RIGHT_INVERTED=true, 往前推车):
- 左电机: 电气正转 → 0x6063 减少 → -N
- 右电机: 物理翻转 → 电气反转 → 0x6063 增加 → +N
- arc_l = -N*const, arc_r_elec = +N*const, arc_r = -(+N*const) = -N*const
- **d_linear = (-N + -N)/2 = 负 → 后退** ← 用户看到的 bug

## 改动 (hls_canopen)
| 文件 | 改动 |
|------|------|
| src/kinematics.rs | `position_delta_to_displacement`: 编码器 delta 取负 (arc_l/arc_r_elec 前加 `-`) |
| src/kinematics.rs | 更新 3 个差分测试用例的输入符号和注释 |

```rust
// 旧:
let arc_l = d_left_counts as f64 * RAD_PER_COUNT * wheel_radius;
let arc_r_elec = d_right_counts as f64 * RAD_PER_COUNT * wheel_radius;

// 新:
let arc_l = -(d_left_counts as f64) * RAD_PER_COUNT * wheel_radius;
let arc_r_elec = -(d_right_counts as f64) * RAD_PER_COUNT * wheel_radius;
```

**不影响**:
- 电机 RPM 命令 (0x60FF) 方向不受影响 — 这是编码器方向问题，不是电机驱动方向
- DiffDrive::inverse — cmd_vel → RPM 不受影响
- RAD_PER_COUNT = TAU/5600 不变 (只是加了个负号在 delta 上)

## 验证
- cargo test: 35 tests pass
- cargo build --release: OK
- pm2 restart hls_canopen + canopen_ros2: done

## 回滚
```rust
// 去掉 arc_l/arc_r_elec 前的负号即可
let arc_l = d_left_counts as f64 * RAD_PER_COUNT * wheel_radius;
let arc_r_elec = d_right_counts as f64 * RAD_PER_COUNT * wheel_radius;
```

# 行人/障碍物容忍度提升 (2026-06-12)

## 现象
一旦有路人侵入 AGV 路线，导航立刻停止并取消任务，不做等待也不尝试绕路。

## 根因
三个串联的「投降太快」参数：

| 参数 | 旧值 | 后果 |
|------|------|------|
| `controller_server.failure_tolerance` | `0.3` | 控制器 0.3s 内找不到可行轨迹就判失败 |
| `progress_checker.movement_time_allowance` | `10.0` | 进度检测 10s 没位移就触发 recovery→abort |
| `planner_server.GridBased.max_planning_time` | `5.0` | recovery 阶段重规划只给 5s 超时 |

`failure_tolerance=0.3` 是主凶：20Hz 控制频率下仅 6 个周期无解就放弃。

## nav2_params.yaml 改动

| 参数 | 旧值 | 新值 | 说明 |
|------|------|------|------|
| `controller_server.failure_tolerance` | `0.3` | `3.0` | 控制器容忍 3s 无解，行人短暂挡路不放弃 |
| `progress_checker.movement_time_allowance` | `10.0` | `20.0` | 让路时间翻倍，避免 10s 没移动判卡死 |
| `planner_server.GridBased.max_planning_time` | `5.0` | `8.0` | 重规划绕路时给更多搜索时间 |

## 预期行为
行人挡路 → 控制器等待最多 3s → 若仍在则触发 recovery (spin/backup/wait) → 全局重规划绕路 → 继续走。
整个过程可能撑数十秒，不会秒退。

## ⚠️ 副作用
- 如果路径真的被**永久障碍物**堵死（如墙、货架），失败回馈会从秒级变分钟级
- 对交付场景（行人频繁），容忍度提高是正确方向；对固定结构环境，这些值偏大
- 可通过 behavior tree 进一步定制（如 wait 行为优先于 backup）

## 后续恢复
```yaml
failure_tolerance: 0.3
movement_time_allowance: 10.0
max_planning_time: 5.0
```
