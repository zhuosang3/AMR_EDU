死锁机制与解决方案

蓝色海湾 AGV 调度仿真引擎 — 死锁分析文档

场景：13台T4无人配送车，1978户小区，无4G网络，LORA低带宽离线通信

图：AGV无人配送车离线死锁机制与解决方案总览

一、对头死锁（Head-on Deadlock）

场景

两辆AGV在同一条窄路（容量=1，无法会车）上相向行驶，谁都无法通过，也无法倒车避让。

  AGV-1 →                          ← AGV-2
  ┌──────┐                        ┌──────┐
  │ 载货  │══════════════════════│ 空车  │
  └──────┘    窄路(容量=1)       └──────┘
              ← 4米宽，只能过一辆 →

蓝色海湾实例：两栋楼之间的连接小道，宽度仅容一车。一辆载货车从3栋去5栋，另一辆空车从5栋回驿站，在小道中段相遇。

检测方式

条件：同一路段(edge)上存在两辆AGV，行驶方向相反，且该路段 capacity == 1

算法伪代码：

fn detect_headon(map: &Map, agvs: &[Agv]) -> Vec<Deadlock> {
    let mut deadlocks = vec![];
    for edge in map.edges() {
        if edge.capacity > 1 { continue; }  // 宽路不会对头死锁
        let agvs_on_edge: Vec<&Agv> = agvs.iter()
            .filter(|a| a.current_edge == edge.id)
            .collect();
        if agvs_on_edge.len() < 2 { continue; }
        // 检查是否有反向行驶的
        let has_forward = agvs_on_edge.iter().any(|a| a.direction == edge.direction);
        let has_reverse = agvs_on_edge.iter().any(|a| a.direction != edge.direction);
        if has_forward && has_reverse {
            deadlocks.push(Deadlock::Headon { edge_id: edge.id, agvs: agvs_on_edge });
        }
    }
    deadlocks
}

解决方案

方案 原理 优点 缺点
路段预约制 AGV出发前预定整条路径的所有窄路段，后续AGV不能使用已被预定的路段 绝对不死锁 路段利用率极低，吞吐量差
滑动窗口预约 只预定前方N段（如3段），走完释放、再预定新的 比全路段预约更高效 窗口重叠时仍可能死锁
优先级协商 通过LORA广播意图，低优先级AGV倒车到最近可会让点 去中心化，接近真实离线场景 需要地图上有可会让点；倒车路径规划复杂
单向化改造 把易堵窄路在地图层面设为单向 从根源消除对头可能 可能增加绕路距离

推荐策略：优先级协商 + 地图层面标注会车点。AGV在进入窄路前通过LORA广播意图，如果检测到对向来车且自己优先级低，在会车点等待。

二、路口死锁（Intersection Deadlock）

场景

多台AGV同时到达一个十字路口（或T字路口），每台车都在等别人先过，结果谁都过不去。

         AGV-B
          ↓
    ──────┬──────
    AGV-C → │ ← AGV-D    ← 四台车同时到路口
    ──────┴──────
          ↑
         AGV-A

  都在等：你先过 → 没人过 → 死锁

蓝色海湾实例：小区中心十字路口，四台车分别从四个方向同时到达。没有4G信号、没有红绿灯、没有中心调度，每台车都”礼让”等待，结果全堵。

检测方式

条件：路口节点(node)的每条入边(incoming edge)上都有AGV处于等待状态

算法伪代码：

fn detect_intersection(map: &Map, agvs: &[Agv]) -> Vec<Deadlock> {
    let mut deadlocks = vec![];
    for node in map.intersections() {
        let incoming: Vec<&Edge> = map.incoming_edges(node.id);
        let all_blocked = incoming.iter().all(|e| {
            agvs.iter().any(|a| a.current_edge == e.id && a.state == AgvState::Waiting)
        });
        if all_blocked && incoming.len() >= 2 {
            deadlocks.push(Deadlock::Intersection { node_id: node.id, agvs: waiting_agvs });
        }
    }
    deadlocks
}

解决方案

方案 原理 优点 缺点
路口互斥锁 把路口当作临界资源，同时只允许一台AGV通过 简单可靠 降低路口吞吐量
路段预约制 把路口作为特殊节点纳入路径预约 绝对不死锁 同对头死锁方案B的缺点
优先级+先到先过 到达时间戳最早的AGV优先通过；同时到达则按载货>空车>回充排序 无需中心协调 需要LORA同步时钟（近似即可）
虚拟信号灯 给路口分配固定时间片轮转（如东-西3秒，南-北3秒） 公平，实现简单 LORA同步精度要求较高

推荐策略：优先级+先到先过。每台AGV通过LORA广播”我到路口了 + 时间戳 + 优先级”，所有等待车按（时间戳, 优先级）排序，最优先的通过，其余等待。

三、环形等待（Circular Wait）

场景

A等B释放路段2，B等C释放路段3，C等A释放路段1，形成闭环——与操作系统教科书级的资源死锁完全同构。

         AGV-A ──→ 等AGV-B让出路段2
          ↑              ↓
    路段3  │              │  路段1
          │              ↓
         AGV-C ←── 等AGV-C让出路段3 ← AGV-B

蓝色海湾实例：三台AGV分别从驿站出发去6栋、7栋、8栋（三栋楼呈三角形排列），每台车占了一条楼前窄路后等待进入下一条，恰好形成闭环。

检测方式

构建等待图(Wait-For Graph):
  - 节点 = AGV
  - 有向边 A→B 表示 "AGV-A 在等待 AGV-B 释放其占用的路段"
  - 检测图中是否存在环 → 存在环 = 环形死锁

算法：DFS染色法检测有向图环，时间复杂度 O(V+E)。

fn detect_circular_wait(map: &Map, agvs: &[Agv]) -> Vec<Deadlock> {
    // 1. 构建等待图
    let mut wait_graph: HashMap<AgvId, Vec<AgvId>> = HashMap::new();
    for agv in agvs {
        if agv.state != AgvState::Waiting { continue; }
        let next_edge = agv.path.next_edge();
        if let Some(blocker) = find_agv_on_edge(agvs, next_edge) {
            wait_graph.entry(agv.id).or_default().push(blocker.id);
        }
    }
    // 2. DFS找环
    find_cycles(&wait_graph).into_iter()
        .map(|cycle| Deadlock::CircularWait { cycle })
        .collect()
}

解决方案

方案 原理 优点 缺点
死锁检测+恢复 周期性检测等待图环，检测到后强制选一台AGV放弃当前路径、重路由 不牺牲正常吞吐量 恢复成本（放弃任务、绕路）
资源排序（全序） 给所有路段编号（1,2,3…），规定AGV只能按升序申请路段；如果下一段编号 < 当前段编号，必须先释放当前段 从根源消除环形等待 限制路径规划自由度
优先级抢占 高优先级AGV可以”抢断”低优先级AGV占用的路段（低优先级AGV被踢出、重路由） 结合业务优先级 被抢占的AGV有额外成本
超时放弃 AGV等待超过阈值（如30秒）自动放弃当前路径，重新A*规划 实现最简单 可能反复超时,造成活锁

推荐策略：优先级抢占。载货车优先级高于空车，空车高于回充车。检测到环形等待时，环中优先级最低的AGV被强制放弃当前路段，退回到上一个节点重新规划路径。这保证了重要任务（载货配送）不受影响。

四、死胡同陷阱（Dead End Trap）

场景

AGV进入一条只有单出入口的断头路完成配送，返回时发现出口被另一辆AGV堵住了。

             驿站/主路
                │
      出口被堵 ← AGV-B停在这里（正在充电/等待任务）
                │
    ════════════╝
    │  死胡同   │
    │  AGV-A    │  ← 送完3栋的件，想出来
    │  已被困   │    但唯一出口被AGV-B堵死
    └───────────┘

蓝色海湾实例：某栋楼门前只有一条路进去，末端是配送点。AGV-A进去送件，此时AGV-B开到路口停下（等任务/等充电/故障），AGV-A送完件想出来，唯一的出口被AGV-B堵死了。

检测方式

条件：节点只有一条边连接（degree == 1 的叶子节点），且那条边的容量已被占满

更精确的检测——AGV进入一段”只有一个出口”的区域后，出口被其他AGV占用：

fn detect_dead_end(map: &Map, agvs: &[Agv]) -> Vec<Deadlock> {
    let mut deadlocks = vec![];
    for agv in agvs {
        let current_node = map.node(agv.current_node);
        // 当前节点是叶子节点（只有一条边进出）
        if map.edges_of(current_node.id).len() == 1 {
            let only_edge = map.edges_of(current_node.id)[0];
            // 检查唯一出口是否被其他AGV占用
            let occupied = agvs.iter().any(|other| {
                other.id != agv.id && other.current_edge == only_edge.id
            });
            if occupied {
                deadlocks.push(Deadlock::DeadEnd {
                    agv_id: agv.id,
                    node_id: current_node.id,
                    blocking_edge: only_edge.id,
                    blocked_by: find_blocker(agvs, only_edge.id),
                });
            }
        }
    }
    deadlocks
}

解决方案

方案 原理 优点 缺点
地图设计避免 在路径规划阶段不允许出现断头路；所有配送点都设计成可环通的 从根源消除 受实际小区道路限制，不一定能做到
进入前检查 AGV进入断头区域前先检查出口是否可用；如果出口被占，在外面等待 简单有效 需要AGV有”预判”能力
路段预约制 进入断头路之前先预定出口路段，确保出来时不会被堵 绝对安全 降低路段利用率
倒车协议 当检测到死胡同时，堵在出口的AGV（如果非载货状态）收到指令主动让开 事后补救 依赖堵路AGV的配合能力

推荐策略：进入前检查 + 路段预约制。AGV的路径规划器识别出”死胡同路段”（度数=1的节点），在进入前检查：如果出口路段当前被占用，在外面等待；如果出口空闲，预定该路段后再进入。这样不会出现”进去了出不来”的情况。

总结对比

死锁类型 核心诱因 检测复杂度 根治方案 实用方案（离线LORA场景）
对头死锁 窄路+对向行驶 O(E) 遍历所有边 路段预约制 优先级协商 + 会车点标注
路口死锁 多车同时抢路口 O(V) 检查路口入边 路口互斥锁 先到先过 + 优先级排序
环形等待 等待链形成闭环 O(V+E) DFS找环 资源全序申请 优先级抢占 + 重路由
死胡同 断头路+出口被堵 O(V) 检查叶子节点 地图设计避免 进入前检查 + 出口预约

关键设计原则

  1. 检测优于预防：仿真引擎必须能检测所有四种死锁，才能在对比实验中量化不同算法的效果
  2. 恢复机制是底线：任何算法都可能有意想不到的死锁场景，必须有兜底的恢复机制（超时重路由）
  3. LORA带宽约束：所有分布式方案必须在”每N秒一条几十字节的广播”这个约束下工作，不能假设实时全量通信
  4. 优先级体系：载货车 > 空车 > 回充车，这个业务优先级贯穿所有死锁解决方案
